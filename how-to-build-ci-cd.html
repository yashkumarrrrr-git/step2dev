How to Build a Modern CI/CD Pipeline (Step-by-Step)

Ship faster and safer: this practical guide walks through designing a robust CI/CD pipeline using GitHub Actions, Docker, artifact storage, and automated Kubernetes deployments — with templates you can copy and run today.

Continuous Integration and Continuous Deployment (CI/CD) are the backbone of modern software delivery. A production-grade CI/CD pipeline automates builds, tests, artifact management, security checks, and safe deployments. Below is a compact, practical pipeline architecture and step-by-step implementation notes you can follow and customize for your projects.

Why CI/CD matters

CI/CD reduces manual steps and human error, shortens release cycles, and raises confidence in shipping changes. For teams delivering cloud-native apps, CI/CD enables:

  • Repeatable, testable builds
  • Automated security and compliance gates
  • Instant rollbacks and safe deployments
  • Faster feedback for developers

Pipeline architecture (recommended)

A modern pipeline typically contains:

  1. Source control (GitHub/GitLab/Bitbucket)
  2. CI system (GitHub Actions, GitLab CI, or Jenkins) for build/test
  3. Containerization using Docker
  4. Artifact registry (Docker Hub, ECR, GCR)
  5. CD system / GitOps (ArgoCD, Flux) for deployments
  6. Observability (Prometheus, Grafana, ELK)

Step 1 — Source control & branching strategy

Use a clear branching model such as main for production, staging (optional), and feature branches for work-in-progress. Protect the main branch with required reviews and passing checks:

  • Require at least one code review
  • Require passing CI checks (lint, unit tests)
  • Use small, focused pull requests

Step 2 — CI: build, test, and static analysis

CI jobs should be fast and reliable. Typical pipeline stages:

  • Install dependencies — cache packages
  • Static analysis — linters, security scanners (e.g., bandit, trivy)
  • Unit & integration tests — run in parallel when possible
  • Build artifacts — create Docker images or build artifacts

Example: GitHub Actions workflow (simplified)

# .github/workflows/ci.yml
name: CI
on:
  pull_request:
  push:
    branches: [ main ]
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v4
        with: {python-version: '3.11'}
      - name: Install deps
        run: pip install -r requirements.txt
      - name: Lint
        run: flake8 .
      - name: Tests
        run: pytest -q
      - name: Build Docker image
        run: docker build -t ghcr.io/yourorg/app:${{ github.sha }} .

Step 3 — Artifact registry & versioning

Push images to a registry with immutable tags (SHA-based). Use separate registries or repos per environment:

  • Tag with sha for traceability
  • Keep latest only for quick testing (avoid relying on it in production)
  • Use repositories with fine-grained permissions (ECR/GCR/GitHub Container Registry)

Step 4 — CD: staging and production deployments

There are two common approaches:

  • Pipeline-driven CD — CI triggers deployment jobs (GNU-style pipelines or scripted deploy)
  • GitOps — push desired state (Helm/Kustomize manifests) to a repo and let ArgoCD/Flux reconcile

GitOps is highly recommended for Kubernetes. Example GitOps flow:

  1. CI writes image tag into a staging manifest in a repo
  2. ArgoCD sees the change and deploys to staging
  3. After smoke tests, update production manifests and let ArgoCD promote

Observability, tests & security gates

Don’t release blind. Add:

  • Automated smoke tests post-deploy
  • Health checks and readiness probes
  • Canary or blue-green deployments for low-risk releases
  • Security scans in CI (SCA, SAST), container scanning (Trivy)

Pipeline templates & snippets

Below is a short sampled snippet to push Docker image to GitHub Container Registry (GHCR):

# Push to GHCR
- name: Log in to GitHub Container Registry
  uses: docker/login-action@v2
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.GITHUB_TOKEN }}

- name: Push image
  run: |
    docker build -t ghcr.io/yourorg/app:${{ github.sha }} .
    docker push ghcr.io/yourorg/app:${{ github.sha }}

Conclusion & next steps

Building a modern CI/CD pipeline is an iterative process: start simple, measure, and evolve. Focus on fast feedback loops, automation of risky steps, and safe deployments (canary/rollbacks). For production workloads, combine GitHub Actions (or your CI), Docker registries, and GitOps tools to run reliable, auditable releases.
